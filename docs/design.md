# Design: Verified Orienteering Platform — permanent flags, timed legs, trusted leaderboards

## Problem Statement

Orienteering on permanent courses today has no trustworthy competition layer. Free tools split the job: MapRun punches controls by GPS only (trivially spoofable), iOrienteering scans QR codes (photographable and shareable), and neither offers per-leg leaderboards or Livelox-style route replay. Livelox and RouteGadget do replay and split analysis, but only for organized events, not for always-open courses you can run any day.

The builder wants: fixed flags in his own forest, an app that times legs between flags, verified leaderboards (fastest total time and fastest per leg), GPS track recording and replay against other runners, course selection and later auto-generation, on both Android and iOS. Reference apps: Livelox, RouteGadget, HittaUt.

## What Makes This Cool

**A leaderboard you can actually trust.** Every existing free tool picks one weak verification method. This design uses two together:

1. **Physical proof at the flag:** each flag carries an NFC tag plus a printed QR code and a human-readable short number as fallbacks. **v1 uses NTAG 213 tags (plain UID read — cheap, zero crypto work); the punch schema carries a `validation_payload` field so upgrading to NTAG 424 DNA (one-time cryptographic SUN codes, unforgeable) is a tag purchase + edge-function addition, not a migration** (decision D20-C, hybrid).
2. **Route proof between flags:** a continuous GPS track must cover the legs; punch times must be consistent with the track.
3. **Time integrity (decision D19):** punch *presence* is proven by the tap; punch *time* is client-asserted, so the server additionally (a) bounds every run's elapsed time inside the real-world window it can establish from sync anchors, and (b) cross-checks claimed times against other runners' punch submissions around the same flags. GPS is honestly treated as a **plausibility filter** (±10-20 m under canopy), not an independent proof.

A run is **verified** only when all checks hold. Unverified runs still appear, but visibly excluded from rankings. On top of that trust layer: per-leg records (Strava-segments-for-orienteering) and Livelox-style multi-runner replay on a real georeferenced orienteering map. Honest v1 label: *verified ≈ NFC presence + bounded time + GPS plausibility*; the 424 upgrade later hardens presence to cryptographically unforgeable.

## Constraints

- Solo builder, side project for himself and his local orienteering community.
- He mounts and maintains the flags himself (hardware cost and weatherproofing are his).
- Real orienteering map as the background from day one (he can obtain the map for his area); OSM outdoor tiles are the unblocking fallback only.
- Must run on Android AND iOS phones.
- Offline-first is non-negotiable: forest dead zones mean punching, timing, and GPS recording must work with zero signal and sync later.
- v1 scope decision: time trial + leaderboard (accounts and verification real from the start), one forest area.

## Premises (agreed 2026-07-10)

1. MapRun and iOrienteering already do free permanent-course timing; this app's reason to exist is the combination they lack — unforgeable check-ins (crypto NFC) + per-leg leaderboards + Livelox-style replay — plus the builder wants to build it.
2. A "verified" leaderboard entry requires BOTH a punch at each flag (NFC tap **or QR scan — equal trust since 2026-07-12**, user decision: the strict 10 m/1 Hz track proximity check carries the presence proof, so a photographed QR without being there still fails) AND a continuous GPS track across the legs. Manual short-code entry is the lower-trust fallback that demotes the run (or leg) below verified; anything less than track+punches shows as unverified.
3. v1 targets one forest area with the builder's own flags, a georeferenced real orienteering map, accounts + leaderboard, built cross-platform so Android and iOS ship from one codebase.
4. Offline-first: check-ins, timing, and GPS recording work with no signal and sync afterwards.
5. The O-map for the area is obtainable (file + permission); OSM outdoor tiles are the temporary fallback so the project is never blocked on the map.

*(Revision note, /plan-eng-review 2026-07-11: premises 1-2 amended by decision D20-C — v1 ships NTAG 213 UID tags with a 424-ready schema; "unforgeable" check-ins arrive with the 424 upgrade. Known v1 limitation, stated honestly in-app: a technically skilled cheater could clone a 213 tag's UID; accepted at club scale, closed by the upgrade.)*

## Approaches Considered

### Approach A: PWA + Supabase (minimal viable)
One installable web app (React + MapLibre GL, Web NFC / camera QR, Supabase backend). Effort M, Risk Med-High.
Rejected because iOS suspends geolocation when the screen locks (broken tracks on long runs) and Web NFC exists only in Android Chrome — the core loop degrades exactly where it matters.

### Approach B: React Native (Expo) app + Postgres backend (ideal architecture) — CHOSEN
One Expo codebase shipping native Android + iOS apps:
- `expo-location` background GPS (screen-off tracking),
- `react-native-nfc-manager` (NFC on both platforms), vision-camera QR scanning,
- MapLibre React Native rendering the georeferenced O-map as raster tiles,
- SQLite offline queue syncing to Supabase (Postgres + PostGIS) for accounts, tracks, splits, leaderboards.
Effort L (human: ~2-3 months of evenings / CC-assisted: several weeks of sessions — directional, not a commitment; the outside voice correctly flagged the original number as optimistic, and review decisions both added scope (web admin, monorepo, E2E harness) and removed the largest risk item (424 crypto deferred by D20-C)), Risk Low-Med. All components are boring, proven tech; the custom parts are verification logic and O-map georeferencing.

### Approach C: Tags-first, no app (creative/lateral)
NTAG 424 tags encode dynamic URLs; each tap opens a web endpoint that validates the one-time code and logs the punch — zero app install. GPS track comes from the runner's watch (GPX upload), matched server-side. Effort S-M.
Rejected as the main path because it drops the live in-forest experience (map, live splits) the builder explicitly wants; retained as an idea for guest/no-app punching later, since the punch-validation backend is the same.

### Courses (v1 definition)

A course is an **ordered list of flag IDs** (start flag, controls 1..n, finish flag) with a name and the map area it belongs to; the course **length is generated from the flag geometry** (sum of straight-line legs, the orienteering convention — revised 2026-07-12 from an admin-entered field). Order is enforced: a punch only closes the leg to the *next expected* flag; out-of-order punches are recorded but flagged. Courses are authored in the **web admin** (below). Any-order score-orienteering courses are post-v1. *(Revised during /plan-eng-review: the earlier "SQL-seed only" decision was reversed by the user in favor of a v1 web admin — D15.1.)*

### Flag registry, lifecycle & web admin (v1)

- **Registry:** every physical tag is registered in a `tags` table (tag UID, diversified-key reference, provisioning date) and mapped to a `flags` row (position, short number, photo). The mapping is versioned: **replacing a dead/vandalized tag writes a new tag row pointed at the same flag** — run history and leaderboards are untouched because punches reference the flag, not the tag.
- **Flag photo:** each flag stores a photo of its mounting spot, shown in the app to help runners (and the maintainer) find it in terrain.
- **Report missing flag:** a button in the app on each flag creates a maintenance report (flag, reporter, timestamp, optional note). Reports appear in the web admin and put a warning marker on the course map. **Reports never alter verification** (decision D21-A): QR/manual punches always demote the leg regardless of open reports — a runner-controllable exception would be the cheapest bypass in the system. A genuinely dead tag therefore makes its legs `partial` for everyone until the tag is replaced; the warning marker plus the demotion *reason* keep that honest and explained.
- **Web admin (v1, minimal):** a small web page (same Supabase backend, admin role via RLS) to: create/edit flags (place on map, photo upload, short code), create/edit courses (pick flags, drag to order), view/resolve missing-flag reports, and follow the tag provisioning/replacement checklist. Runner-facing course browsing stays in the app; the admin is desktop-first.

**Leaderboard semantics (v1):** each leaderboard (total-time and per-leg) **ranks** each user's best verified run/leg only; ties broken by earlier completion. Partial and unverified runs appear on the leaderboard screen below the ranked list, visibly marked and unranked (per the approved wireframe — the trust story made visible). DNF/abandoned runs appear only in the runner's personal history. **Class filter chips** (P7-D13-A): Overall · M · W · U14 · U18 · O40 · O60 — a class view ranks the same data filtered by the runner's class at run date; class records derive from the filter, no separate tables.

### Write authority (v1 — security boundary)

Clients may only **insert raw sync payloads** (runs, punch events, track points) into their own rows, enforced by Supabase RLS. All derived truth — punch validation results, leg statuses, run status, leg_splits, leaderboard rows — is computed **exclusively** by the verification edge function running with service-role privileges. Status and split columns are immutable to client tokens: RLS restricts *rows* (no UPDATE policies), and **Postgres column grants restrict *columns*** — without them a client could INSERT its own run with `status='verified'` pre-set, or UPDATE `profiles.is_admin` on its own row (both found and closed in the anonymous-sign-in RLS review, migration 0008). A client can therefore fabricate raw data but can never mint `verified`; fabricated raw data still fails SUN/CMAC validation server-side. (Added by /plan-eng-review, decision 1A; column grants added 2026-07-12.)

**Shared verification-core (decision 3A):** the status enums, leg/run trust rules, split math, and tuning-constant handling live in one pure-TypeScript package (`packages/verification-core`, no I/O) imported by BOTH the Expo app (live provisional statuses) and the Deno edge function (authoritative statuses). Repo is a workspace monorepo (`apps/mobile`, `apps/admin-web`, `supabase/functions`, `packages/verification-core`). The golden-file oracle tests target this package, covering both consumers at once. Client and server can therefore never disagree on the rules — only on inputs (which sync reconciles).

**Sync idempotency (decision 2A):** the client assigns a UUID to every run and punch at creation; the server upserts on that UUID, so an interrupted sync retried later is a harmless no-op instead of a duplicate. The SUN seen-counter set keys on punch UUID + counter, so a legitimate re-send of the same punch never self-rejects. Partial sync is resumable record-by-record — the right behavior for one-bar forest-edge connectivity.

### Verification rules (v1)

- Punch = NFC tap (v1: NTAG 213 UID read, matched against the tag registry; the punch row carries a nullable `validation_payload` for the future 424 SUN upgrade) or QR scan (**equal trust to NFC since 2026-07-12** — presence is proven by the 10 m/1 Hz proximity check) or manual short-code entry (lower trust, for dead tags). On iOS, Core NFC needs a user-initiated foreground scan session that cannot be pre-armed (~60 s timeout), so the real per-flag flow is: unlock → app is already open on the run screen → tap "punch" to arm the session → tap the tag. There is no background-read shortcut on iOS; friction is equal for all runners. **The leg clock stops at tag read**, not at UI confirmation, so the bottom-sheet animation never costs time.
- **Run start:** the clock starts at the start-flag tag read (not a button press). A manual start punch makes the first leg at best `partial`, per the leg rules below (QR counts as NFC). **The GPS track also records from the start punch on** (user decision 2026-07-12: no pre-start wandering on the route view); the location watcher warms up beforehand without recording, so the start flag still gets samples inside the ±60 s punch proximity window.
- Dedup and replay (corrected during outside-voice review — the two earlier rules used incompatible keys): **idempotency dedupes on punch UUID** (same UUID re-sent = upsert no-op); **replay handling is tag-generation-dependent** — v1 (NTAG 213) has no cryptographic replay protection, so identical-UID punches from different runs are legal by design (that's just two runners tapping the same flag) and time-forgery is countered by the D19 bounds instead; after the 424 upgrade, the server additionally keeps a per-tag set of seen SUN counters and rejects a counter it has seen under a different punch UUID (never a high-water-mark rule — offline sync means counters legitimately arrive out of order).
- **Time-integrity checks (decision D19, server-side):** (a) *elapsed-time bound* — a run's claimed elapsed time must fit inside the wall-clock window the server can establish for it (previous sync anchor / app-start ping when online → sync time); runs that claim less time than physically available are fine, runs claiming a window inconsistent with server-observed anchors are demoted with reason. (b) *Cross-runner consistency* — punch submissions on the same flag from different runners establish coarse physical orderings; a run whose claimed times are inconsistent with surrounding runners' submissions is flagged for review. Fresh install that goes straight to the forest with zero signal: no pre-run anchor exists, so only the post-sync bound applies and the run is capped at `partial` with reason "no pre-run time anchor" (one online app-open before running clears this forever).
- **Key management (post-v1, 424 upgrade):** one master AES key in Supabase secrets, per-tag keys via UID diversification (NXP AN10922); SUN validation (PICCData decryption + AES-CMAC) ported to TypeScript in the edge function with the Python `sdm-backend` as a golden-file test oracle. None of this is v1 work (decision D20-C). This is the other half of the NTAG risk budget (see Open Questions 2): days, not hours, including provisioning.
- Track check: GPS track must pass within tolerance of each punched flag at a time consistent with the punch, and the median-filtered speed over 10-second windows must stay below a tunable ceiling (default 8 m/s) between flags. Raw instantaneous speeds are never used — ±10-20 m canopy multipath produces fake >30 km/h spikes. A leg's track "covers" it when no sample gap exceeds the max-gap constant (default 30 s). Three field-tuning constants total: flag-proximity tolerance (tuning v2, 2026-07-12: **10 m** — user decision, strict against verify-from-a-distance false positives; the app records at **1 Hz** so the check has a dense track; accepted tradeoff: canopy drift may demote honest runs to partial, revisit after the field test), speed ceiling, max track gap.
- **Per-leg status enum:** `verified` (both bounding punches NFC **or QR** — equal since 2026-07-12 —, track covers the leg, speed check passes) / `partial` (manual punch on either end, track gap exceeds max-gap, or clock basis lost during the leg) / `unverified` (a bounding punch missing, or the track fails the proximity/consistency check for the leg — including spoof-shaped failures). Run status = min over its legs (ordering: verified > partial > unverified). Per-leg leaderboard entries require that leg verified; the total-time leaderboard requires all legs verified.
- Run status is therefore exhaustive by construction: any run maps to the min of its leg statuses; a run with no punches at all (the GPS-only-track spoof) or no track has every leg `unverified` and is `unverified`. Status reasons are stored and shown ("leg 3: track gap 74 s").
- **Re-runnable verification (decision D22-A):** statuses are derived data — a pure function of stored raw punches/tracks plus a **versioned tuning config** (proximity tolerance, speed ceiling, max gap live in a config table, not code). Changing the config recomputes statuses for affected runs via the same verification-core functions; every status row records the config version that judged it, and re-judged leaderboard entries show a "re-verified under updated rules" note. Statuses are immutable to *clients*, never to the verification function.
- Clock integrity: elapsed times come from **monotonic device-clock deltas only** — a run must be fully timeable with zero signal from start to finish (airplane-mode test). Server anchoring is opportunistic: last-known clock offset before the run plus a fresh anchor at sync, used to place the run on the wall-clock timeline and cross-check punch plausibility, never to compute elapsed time. Wall-clock is never trusted. If the monotonic basis is lost mid-run (reboot, app kill), affected legs demote to `partial`.
- Post-sync feedback: if a punch fails server validation after an offline run, the finish/history screen states it explicitly ("punch at flag 3 failed verification — run marked unverified") rather than silently demoting. A failed SUN/CMAC check is spoof-shaped and maps to `unverified` for that leg, per the enum above. The runner's data is never deleted.

### Device failure matrix (v1 — decision 4A)

Pre-run gate first (4C folded in): the start screen blocks until NFC is on, location permission is "Always/While using" with precise accuracy, battery optimization is exempted (Android), and storage has headroom. Then, defined behavior for mid-run failures — never a crash, never silent loss; the local run log is always preserved:

| Failure | Runner sees | Recorded | Resulting status |
|---|---|---|---|
| NFC toggled off at a flag (Android) | "NFC is off" sheet with settings shortcut + QR fallback offered | punch via QR if used | leg `partial` (QR rule) |
| NFC read fails 3× (damaged tag) | "Try QR or enter flag number" + report-missing-flag button | QR/manual punch + optional report | leg `partial`; flag report opened |
| Camera permission denied on QR fallback | manual short-code entry offered | manual punch | leg `partial` (manual rule) |
| Location permission revoked mid-run | persistent warning banner at next foreground | track ends at revocation | legs after cut demote per track-gap rule |
| OS/battery-saver kills background GPS | warning at next app foreground ("tracking gap Xs") | gap in track | affected legs per track-gap rule |
| App force-quit / crash mid-run | on relaunch: "Resume run?" restores from SQLite log | punches + track up to kill survive (append-only) | clock-basis rule applies (legs during dead window `partial`) |
| Phone reboot mid-run | same resume flow | same | monotonic basis lost → affected legs `partial` |
| Storage full during track write | pre-run gate warns; mid-run: stop track writes, keep punches | punches only from that point | legs after cut per track-gap rule |

Every row above becomes at least one test case (see test plan).

Home shows: last run summary, sync status, "Run a course" primary action. Course detail is where records, flag warnings (open reports), and the START affordance live. Flag photos appear in course detail and in the punch-fallback sheet.

### Punch interaction (decision P2-5A — the core interaction, fully specified)

- **Android:** NFC auto-arms whenever the run screen is foregrounded (reader mode) — hold phone to flag, done, zero taps. **iOS:** full-width bottom-zone **PUNCH button ≥72 pt tall**, thumb-reachable one-handed, which arms a Core NFC session.
- States: `idle` ("PUNCH — flag #4") → `arming` → `scanning` ("Hold phone to the flag", ~60 s countdown ring) → `success` (green full-screen flash + strong haptic + leg-split bottom sheet, auto-dismisses after 8 s, swipe-dismissable, never blocks the map longer) → `read failed` (retry; after 3 fails escalate to "Use QR / enter flag number" sheet showing the flag's photo).
- Wrong/out-of-order flag: orange feedback "That's flag 7 — next is #4", punch recorded + flagged, run continues.
- The leg clock stops at tag read (already specified); the success sheet is feedback, not part of the timed path. Platform handling-time bias (Android faster) is accepted and quantified by the T9 friction field test; per-leg noise from navigation dominates it.

### Onboarding — "set up at home, run tomorrow" (decision P3-7A)

Four screens, all completable at home with signal; the trailhead then requires zero signal and zero reading:
1. **Welcome + profile:** no visible sign-in at all (anonymous device account created silently), framed with the one-line promise ("Timed courses in [forest name]. Verified leaderboards."). The runner enters display name, email (unverified — no code, no link), birth year and gender, with the honest reason shown ("so you compete in your class — Elite and age classes, M and W ranked separately") — one tap and they're in. A **sign-in shortcut** ("Already set up on this phone?") skips sign-up, anchored and permissions when the device already has an account with a profile (2026-07-12).
2. **Anchored:** after first online open — "You're anchored — your runs will count." (This is the designed face of the no-pre-run-anchor rule; it turns a hidden penalty into a completed step.)
3. **Permission gate, one honest line each:** Location Always — "so your track records with the screen off" · NFC — "tapping flags proves you were there" · Battery exemption (Android) — "so the system doesn't kill tracking mid-run" · Notifications (optional) — "sync results when they're ready". Each with its OS prompt fired only after the why-copy.
4. **Course browse as reward:** land on the course list with photos — "pick tomorrow's run."

### Classes & run conventions (decision P7-D13-A — reverses the earlier class deferral at the user's request)

- **Classes (v1):** profile collects birth year + gender at onboarding (step 1 gains two fields). Class = derived at run date: M/W × {U14, U18, Elite, O40, O60} — "Elite" is the open adult class (orienteering's 21E; revised from "open" at the user's request 2026-07-12). Leaderboards get filter chips with **every class separately per gender**: Overall · M · W · M Elite · W Elite · M U14 · W U14 · M U18 · W U18 · M O40 · W O40 · M O60 · W O60; class records are simply the fastest verified run within the filter; the "you" row rules apply inside class views; empty classes invite ("be the first M-O40 name here"). The course detail screen shows, before the start, the viewer's own class with its best and average verified time on that course ("M Elite · best / avg", empty class: "be the first") — added 2026-07-12.
- **vs-best (mid-run):** compares against **your class record** when one exists, else the overall course record — whichever reference is used is cached on the phone at run start (offline-safe) and named in the split sheet ("0:12 faster than the M-O40 record pace"). Finish screen additionally shows personal best.
- **Finish:** tapping the finish flag ends the run instantly — no confirmation (the clock stopped at tag read; a dialog adds only fat-finger risk).
- **DNF:** long-press the elapsed timer → "Abandon run?" → saved to private history as DNF, never on leaderboards.

### Run-screen navigation philosophy (decision P7-D12 — user's own design, sport-pure)

- **No live position dot.** GPS records silently in the background for verification and later replay; the runner never sees their own position on the map mid-run. Navigation is done with the map and your head — that IS the sport. (The wireframe's blue dot and trail are removed; the "~230 m NE" bearing assist is likewise cut.)
- **Small compass rose in a corner, togglable** on/off from the run screen.
- **Map rotation: manual gestures only.** Two modes: free-rotate (pinch/twist) and **north-lock** (map locked north-up relative to the phone). **No heading-based auto-rotate, ever.**
- Consequence: the run screen shows map + course overlay + elapsed time + punch affordance + compass. Nothing else. This is stronger differentiation than any feature: it's paper-map orienteering with digital timing and proof.

### Design tokens (decision P5-11A — the plan's design system)

- **Type:** IBM Plex Sans (UI) + IBM Plex Mono (all times/numbers, tabular figures). Elapsed timer 48 pt bold mono · next-control 24 pt semibold · body 17 pt · floor 16 pt. No system font stacks.
- **Color:** the O-map is sacred — ISOM colors never tinted or overlaid with translucency. UI chrome: solid `#141414` panels on white. Accent = ISOM magenta `#D10F7C` (course overlay AND primary actions). Verified `#0A7A0A` · warning `#B45309` · error `#B91C1C`.
- **Contrast:** in-run text ≥7:1 (AAA — direct-sunlight budget); all body text ≥4.5:1. Panels over the map are always solid.
- **Touch:** ≥60 pt targets on the run screen (wet/gloved hands), ≥44 pt elsewhere.
- **A11y floor:** VoiceOver/TalkBack labels on leaderboard, history, course screens; the run screen deliberately prioritizes glanceability and haptics over screen-reader flows.
- Typeface is swappable on taste; the scale, contrast, and target numbers are the contract.

### Browse map, run history & fair-start rules (user decisions 2026-07-12)

- **Re-run cooldown (ranking rule, verification-core):** repeating a course within 7 days of your last run on it keeps the new run UNRANKED — route knowledge is fresh, an immediate repeat would be an unfair time. Cooldown counts from the last rank-eligible run; the run itself is stored, shows in history and on the finish screen. The map popup warns ("Run again · unranked this week").
- **Fair start:** before the start punch, both the course-detail map and the run screen show ONLY the start flag (triangle pointing north) — the course reveals at the start punch, like getting the map at the start signal. The GPS watcher warms up pre-start but the track records from the start punch.
- **Course picker:** already-run courses render grey ("done") instead of their difficulty color; their popup shows your best time and asks: Run again (with cooldown note) or View my run. Courses sharing a start point collapse into ONE pin with a count badge whose popup lists them all (stacked pins would hide each other). A find-me button recenters on the live position.
- **Run history ("My runs"):** bottom toggle bar on the browse map slides up a list of all own runs — course, date, total time, trust status (or "pending sync" / "abandoned") — with status filter chips (all · verified · partial · unverified). Tapping a run (or "View my run" on the map) opens the **run detail page**: route over the course map on the top half; below it start/finish wall time + total, and the per-leg table with split, cumulative time at each control, and comparison vs the course's BEST and AVERAGE verified split (green = at/under). Runs are STORED server-side (runs/leg_splits/tracks rows on the account); the route additionally caches ON DEVICE at the finish (newest 20 runs, works offline/pre-sync) with the server track as fallback — reading tracks is allowed for the runner's OWN runs only (policy `read_own_tracks`, 0013; other people's movement data stays private).
- **Profile settings (gear, top-right on the browse map):** edit display name, birth year, gender. The email is shown READ-ONLY as a hint (set once at onboarding; own address comes via the `get_own_email()` RPC, 0012 — other emails stay client-unreadable). A profile photo can be taken with the camera; it stays ON DEVICE for now (user hasn't decided on further use — nothing is uploaded). A device-local **"Punch by QR code" toggle** makes the run screen's punch button open the QR camera directly instead of arming NFC (QR = NFC trust since 2026-07-12, so no penalty).

### Leaderboard "you" moment (decision P3-8A)

Own row: accent fill + "you" tag. If own rank is off-screen, pin the own-row at the bottom edge of the list viewport until scrolled to. Finish screen (post-sync) deep-links: "See yourself on the leaderboard →". This is the product's emotional payoff; it gets designed, not defaulted.

### Finish screen — offline/pre-sync vs post-sync (decision P2-6B)

The wireframed finish screen is the POST-SYNC state. The v1 happy path (airplane-mode run) ends on the PRE-SYNC state, which must be designed separately:
- **Pre-sync:** splits + total time computed from local data; verification shown as **"provisionally verified — syncing…"** chip (never a bare ✓ — the app must not claim server truth it doesn't have); ranks render as "— (pending sync)". When connectivity returns, the screen upgrades in place.
- **Post-sync:** ranks fill in, badge resolves to verified/partial/unverified with its reason; a post-sync **demotion gets a designed slot** on this screen and in My runs ("punch at flag 3 failed verification — run marked unverified"), not a toast.

## Test Plan (decision 5B + e2e-mobile)

Scope decision: automated coverage concentrates on the trust surface (verification-core, crypto, sync, RLS) plus E2E user journeys via [e2e-mobile](https://github.com/trinhngocthuyen/e2e-mobile) (Python, Appium + Pytest, write-once for iOS/Android, page-object style). UI polish and admin flows stay manual in v1.

**Frameworks:** Vitest (`packages/verification-core`, admin web logic) · Deno test / Vitest (edge function) · Pytest + e2e-mobile (mobile E2E) · Python `sdm-backend` as golden-file oracle.

1. **verification-core (Vitest, exhaustive — pure functions):** every `legStatus` branch (verified; partial via QR/manual punch, track gap, lost clock basis; unverified via missing punch, proximity/speed failure), `runStatus` min-ordering, split math from monotonic deltas, vs-best comparison, median-filter speed windows, boundary values for all three tuning constants (e.g., gap exactly 30 s).
2. **Punch validation (v1) + time integrity:** UID-to-registry matching (unknown UID → clean reject with reason), tag-replacement continuity (new UID, same flag, history intact), elapsed-time bound checks (with/without pre-run anchor), cross-runner consistency flagging, fresh-install-no-anchor → capped at partial. *(The SUN golden-file oracle corpus moves to the 424 upgrade, per D20-C.)*
3. **Sync + write authority (integration, local Supabase):** re-send of synced run is a no-op (idempotency); interrupted-then-retried sync converges; client token CANNOT insert/update status columns, other users' rows, or leg_splits (RLS proof); punch failing CMAC yields the user-visible reason string.
4. **Mobile E2E (e2e-mobile; NFC via debug mock punch provider):** full run happy path (select course → punch all → finish → splits); full offline run (airplane mode start→finish, sync later, statuses correct); resume after app kill mid-run; double-tap same flag; the three spoof scenarios end at not-verified.
5. **Manual for v1 (accepted 5B tradeoff):** admin web flows (course authoring, tag replacement, report inbox), run-screen visual states, and the sensor-level rows of the device failure matrix (NFC off, permission revocation on-device) — smoke-checked on real devices at field-test time. Logic-level consequences of those rows (status demotions) ARE automated in layer 1.
